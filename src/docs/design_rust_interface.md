# Rust Interface Design (Oculomotor)

This document outlines how the Rust `Oculomotor` component drives the eBPF sampler.

## 1. Lifecycle Management

The `Oculomotor` struct will own the `SamplerSkel` (skeleton) generated by `libbpf-rs`.

```rust
pub struct Oculomotor {
    skel: SamplerSkel<'static>,
    ringbuf: RingBuffer<'static>,
    // ...
}
```

### Initialization (`builder()`)
1.  **Open**: `SamplerSkelBuilder::default().open()?`
    *   This parses the object file but doesn't load it into the kernel yet.
2.  **Initial Config**: Set `target_pid` and `min_sample_interval_ns` on `skel.maps.bss_data`.
    *   *Crucial*: `target_pid` must be known before attachment if we want to filter from the start (though our current BPF logic allows dynamic updates, setting it early is cleaner).
3.  **Load**: `skel.load()?`
    *   Verifies and loads programs into the kernel.
4.  **Attach**: `skel.attach()?`
    *   Attaches `sched_switch` and `perf_event` hooks.

## 2. Configuration Interface

We expose methods to update the "Control Channel" (Global Variables).

```rust
impl Oculomotor {
    pub fn set_target_pid(&mut self, pid: u32) {
        self.skel.maps_mut().data().target_pid = pid;
    }

    pub fn set_sample_rate(&mut self, interval_ns: u64) {
        self.skel.maps_mut().data().min_sample_interval_ns = interval_ns;
    }
}
```

## 3. Hardware Counter Management

This is the most complex part. The `PERF_EVENT_ARRAY` map (`counters`) expects File Descriptors (FDs) from `perf_event_open`.

### The `update_counters` method
When the `Scheduler` requests a new set of events:

1.  **Teardown**: Close existing `perf_event` FDs (if any) to release hardware counters for all CPUs.
2.  **Setup**: For each requested event (up to `MAX_COUNTERS`):
    *   Iterate over all online CPUs.
    *   Call `perf_event_open` for the specific CPU.
3.  **Map Update**: 
    *   The BPF map `counters` is a `PERF_EVENT_ARRAY`. Keys `0` through `MAX_COUNTERS-1` correspond to the logical counter slots used by the BPF program.
    *   For each slot `i` (key) and each CPU `c`:
        *   Update the map using `bpf_map_update_elem` with `key = i` and `value = fd`, enabling `BPF_F_CURRENT_CPU` logic or explicitly setting it for the target CPU if utilizing `libbpf` headers correctly (or simply calling update from the correct thread, though in Rust `libbpf-rs` handles this via `Map::update` which might update for *all* CPUs or specific ones depending on flags).
    *   *Implementation Note*: We use `libbpf_rs::Map::update` with `MapFlags::ANY`. Since `PERF_EVENT_ARRAY` is a special per-cpu map where the kernel associates the FD with the CPU of the calling process (or specified via flags in low-level usage), we must ensure we populate it correctly. 
    *   *Simpler Approach*: We open the perf event for **every CPU**, and then for each slot index (0-3), we set the FD for that CPU. `libbpf-rs` abstracts this. We will ensure that for key `K`, there is a valid FD for every CPU.

**Summary**: The user space driver ensures that keys `0..3` in the `counters` map are populated with valid perf event FDs for all CPUs. The BPF program simply calls `bpf_perf_event_read_value(&counters, i)` which automatically retrieves the counter value for the CPU it is currently running on.

## Summary of Flow

1.  **User**: `saccade run --target-pid 123`
2.  **Main**: `Oculomotor::new(123)` -> Loads BPF.
3.  **Main Loop**:
    *   `oculomotore.poll()` -> Returns samples.
    *   `scheduler.next(samples)` -> Returns `active_events`.
    *   `oculomotor.set_counters(active_events)` ->
        *   Closes old FDs.
        *   Opens new FDs (per CPU).
        *   Updates BPF map slots 0..3.
